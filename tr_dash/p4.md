## 1) Problem Statement + Objectives + Risks + Assumptions

### Problem Statement

현재 화면은 **Shift Brief + KPI + Alerts + Voyages + Timeline + Gantt + History/Evidence + Export/Readiness**까지 한 화면에 담아 **“운영 콕핏”과 “분석 워크벤치”가 한 레이어에서 경쟁**하고 있습니다. (상단에 Live/History/Approved/Compare, 리스크 필터, Location/Schedule/Verification 섹션과 동시에 Schedule Timeline + Gantt + History/Evidence가 동일 스크롤 축에서 공존) ([trdash.vercel.app][1])
이 구조는 기능은 풍부하지만, **시간 민감한 의사결정(Go/No-Go, 48h 핵심 작업, 일정 변경 승인)**에서 사용자가 “지금 무엇을 해야 하는지”를 빠르게 확정하기 어렵게 만들 가능성이 큽니다.

> NN/g는 대시보드를 “한 화면에서, 최소 상호작용으로, 빠르게 행동 가능한 정보를 전달하는 것”으로 정의하며(특히 operational dashboard는 time-sensitive), 과도한 탐색/분석 목적 UI와 섞이면 목적이 흐려지기 쉽습니다. ([Nielsen Norman Group][2])

### Objectives (KPI 관점)

운영(Shift) 상황에 맞는 목표를 **측정 가능한 형태**로 고정합니다.

* **Time-to-Decision**: “Shift Brief 진입 → 오늘의 Go/No-Go 확정/승인”까지 걸린 시간(분)
* **Task Success Rate**: “48시간 내 중요 작업 확인 → 담당/마감/근거 확인 완료” 성공률(%)
* **Error Rate**: 일정 변경(단건/벌크) 후 **충돌(conflict)** 혹은 **재작업(undo/redo)** 발생률(%)
* **Recovery Success**: 충돌/잘못된 변경 발생 시 **preview→수정→승인→반영**으로 복구 성공률(%)
* **A11y Blockers**: WCAG 2.2 AA 차단 이슈(건)
* **p95 Interaction Latency**: Gantt/Timeline 핵심 인터랙션(p95) ms

(사용성 정의를 effectiveness/efficiency/satisfaction로 보는 ISO 9241-11 정의는 위 KPI 체계화의 근거로 적합합니다.) ([W3C][3])

### Risks

* **의사결정 리스크**: 정보가 “많다”가 아니라 **결정에 필요한 근거/상태/다음 행동이 한 덩어리로 묶이지 않으면** 결정이 늦거나 잘못될 위험.
* **데이터 신뢰 리스크**: “갱신일/주기”가 일부 영역에만 노출(예: 기상·해상 리스크 최종 갱신/주간)되어, 사용자가 화면 전체의 최신성을 직관적으로 판단하기 어려움. ([trdash.vercel.app][1])
* **시간대(UTC/Local) 리스크**: 상단에 “날짜(UTC)”가 있으나, Shift Brief에 “다음 결정 창: 18:00 현지”가 함께 존재 → 동일 화면 내 time model이 섞일 때 오해 가능. ([trdash.vercel.app][1])
* **접근성/안전 리스크**: 드래그 기반(Gantt 이동 등)과 포커스/타깃/일관된 도움말 등이 WCAG 2.2 신규 SC에서 차단 이슈가 되기 쉬움. ([W3C][4])
* **성능 리스크**: “Loading Gantt…” 상태가 노출 → 초기 로딩/상호작용 지연 시 운영 화면으로서 치명적일 수 있음. ([trdash.vercel.app][1])

### Assumptions (명시)

* 주 사용자는 **데스크탑**에서 교대(Shift) 단위로 본다.
* 핵심 작업은 **Go/No-Go 판단 + 일정 조정 + 근거/증빙 기록 + 보고서 Export**다. ([trdash.vercel.app][1])
* 화면은 **영/한 혼용**을 의도했으나(라벨에 한국어+영어 병기), 현재는 일부 중복/비일관이 있다(예: “항차 항차 선택”). ([trdash.vercel.app][1])

## 2) Personas(2–3) + Validation Qs

### Persona A — Shift Lead (Ops)

* 목표: **오늘/이번 교대의 “결정(Go/No-Go)”**과 **즉시 조치(48h tasks)**를 확정하고, 다음 교대에 인수인계.
* 필요: “결정 근거(기상/해상/허가/자원) + 최신성 + 담당/마감 + 변경 내역”이 한 눈에.

**Validation Qs**

1. 사용자가 30초 내에 “오늘의 Go/No-Go 상태/근거/갱신 시각”을 말할 수 있는가? ([trdash.vercel.app][1])
2. 1분 내에 “48h 중요 작업 8건”의 **우선순위/담당/막힘(Blocking) 원인**을 특정할 수 있는가? ([trdash.vercel.app][1])

### Persona B — Planner/Scheduler

* 목표: 일정 변경(단건/벌크) 수행 후 **충돌 없는 반영** + **변경 요약/승인/되돌리기**.
* 필요: Preview/Compare/Conflict detection, 변경 범위(patch/full), 승인 흐름. ([trdash.vercel.app][1])

**Validation Qs**

1. “단건 변경 → preview → conflict 해결 → 승인 → 반영”을 3분 안에 완료 가능한가?
2. 변경 후 “무엇이/왜/어떤 근거로 바뀌었는지”를 MD/JSON export로 바로 공유 가능한가? ([GitHub][5])

### Persona C — HSE/Compliance / QA

* 목표: 안전/허가/증빙 관점에서 “결정 이력/근거 링크”가 **append-only**로 남고 재현 가능해야 함. ([trdash.vercel.app][1])

**Validation Qs**

1. 특정 날짜/항차에서 “의사결정 노트 + 근거 링크 + 변경 diff”를 2클릭 내 찾을 수 있는가? ([trdash.vercel.app][1])

* 기상/항만/자원 변화 → 일정 변경 필요 → Preview/Compare로 영향 확인 → 승인/반영 → 기록/공유 ([trdash.vercel.app][1])

## 4) Flows + Recovery paths + Metrics

### Flow A — Go/No-Go Decision (Operational core)

**현재 화면 근거**

* Alerts triage에 “해상 이송 Go/No-Go / 결정: GO / 태그 / 갱신일”이 있음 ([trdash.vercel.app][1])

**개선 Flow (권장)**

1. **Decision Card**(상단 고정): 상태(GO/NO-GO/HOLD) + 기준(풍속/파고 임계) + 근거 링크 + 최신성 + 담당자
2. “결정 확정” 버튼 → **결정 요약(자동 생성)** → 승인(1~2인) → Notes & Decisions에 자동 기록
3. 변경 발생 시 “무효화/재결정” + diff 생성

**Recovery**

* 잘못된 결정/근거 링크 → “Reopen decision” → 새 버전(append-only)로 남김

**Metrics**

* Time-to-Decision
* Decision Reversal Rate(재결정 비율)

---

### Flow B — Schedule Update (Single/Bulk + Preview + Conflict)

**현재 화면 근거**

* “AGI Schedule Updater: Single/Bulk, Preview”가 있고, 커맨드 팔레트에 “/bulk /conflicts /export mode=patch|full /undo /redo”가 존재 ([trdash.vercel.app][1])
* Schedule Timeline 테이블이 항차별 주요 마일스톤을 나열 ([trdash.vercel.app][1])
* Gantt에 “이동(드래그)”이 명시됨 ([trdash.vercel.app][1])

**개선 Flow (권장)**

1. 변경 입력(단건/벌크/드래그)
2. Preview 패널에서 영향 범위(어떤 항차/마일스톤이 얼마나 밀리는지) + 충돌(자원/허가/기상 창)
3. **Apply Patch**(영향 범위 최소) / Apply Full(전체 재계산) 선택
4. 승인(운영 데이터 변경은 사람이 승인) → 반영 → 자동 기록/Export

> “승인형 실행(approval)”은 에이전트/툴 기반 UX에서 안전 레이어로 표준화되는 추세이며, Vercel AI SDK 6도 tool execution approval(사람 승인)을 핵심 기능으로 제공/설명합니다. ([Vercel][6])

**Recovery**

* 즉시 Undo(가시적 버튼 + 히스토리 스택)
* “Reset to baseline” (Compare baseline 스냅샷 기준)
* 충돌 발견 시 “Resolve” 가이드

**Metrics**

* Conflict Rate
* Recovery Success
* Undo after Apply 비율(품질 경보)

---

### Flow C — Evidence / History / Readiness

**현재 화면 근거**

* “History Evidence Compare Diff Trip Closeout / Load SSOT …” 문구가 존재 ([trdash.vercel.app][1])
* README에 append-only history/evidence(localStorage), Trip report export(MD/JSON), Next trip readiness가 명시 ([GitHub][5])

**개선 Flow (권장)**

* Evidence는 “결정(Decision) 객체”에 종속:

  * Decision ID → 근거 링크(Weather/Permit/Resource) → 첨부(파일/URL) → 책임자 → 타임스탬프
* Readiness는 “다음 Trip” 중심 체크리스트(블로커/담당/예상 해소일)로 재구성

**Metrics**

* Evidence completeness(결정당 근거 링크 보유율)
* Readiness blocker aging(블로커 평균 체류 시간)

## 5) IA + Label strategy

### 핵심 인사이트: “한 화면”이 아니라 “두 속도(2-speed) IA”가 필요

현재는 operational + planning + audit가 같은 스크롤 레벨에 있습니다. ([trdash.vercel.app][1])
NN/g 정의에 맞추면, 이 제품의 1차 목적은 **Operational(시간 민감) 대시보드**로 보입니다. ([Nielsen Norman Group][2])
따라서 IA를 아래처럼 분리하는 것이 효과적입니다.

#### 제안 IA (상단 탭/모드 재정의)

* **Monitor(운영)**: Shift Brief, Decision Center, Alerts, 48h Actions, 최신성/리스크
* **Plan(계획)**: Timeline table, Gantt, Scenario/Compare, Reflow preview, Conflicts
* **Verify(검증/증빙)**: Evidence, Notes & Decisions, Trip Closeout, Readiness
* **Export/Share(공유)**: MD/JSON, 스냅샷 링크, 인수인계 패킷

현재 상단에 Live/History/Approved/Compare가 이미 있으므로, 이걸 “데이터 상태”가 아니라 “사용자 작업 모드”로 승격하는 방향이 자연스럽습니다. ([trdash.vercel.app][1])

### Label strategy (한/영 병기 정리)

* 중복 라벨 제거: 예) “항차 항차 선택” → “항차(Trip) 선택” 한 번만 ([trdash.vercel.app][1])
* 시간 표현 통일:

  * 날짜 입력은 UTC(기본) 유지하되, **모든 시간 표기 옆에 Local 동시 표기**(특히 “결정 창: 18:00 현지”) ([trdash.vercel.app][1])
* 전문 용어 사전(tooltip) 고정:

  * LO/LI, MZP, SPMT, PTW, CERT, COL 등은 범례(legend)에서 “정의 + 의사결정 영향”을 2클릭 이내로(README에도 GanttLegendDrawer 의도가 있음) ([GitHub][5])

## 6) UI Direction + Component Specs + Accessibility Notes

### UI Direction (결정 우선, 근거가 보이는 운영 콕핏)

**현재 강점**

* Shift Brief가 “일일 펄스 / 48h 중요 작업 / 다음 결정 창”을 제공 ([trdash.vercel.app][1])
* Alerts triage에 Go/No-Go, 태그, 갱신일이 존재 ([trdash.vercel.app][1])
* Schedule table로 항차별 마일스톤이 명확 ([trdash.vercel.app][1])

**개선 방향**

1. **Decision Center를 1st viewport(스크롤 없이 보이는 영역)**에 고정
2. “정보”를 “행동 단위”로 묶기

   * (예) Alerts triage → “결정 카드 + 승인 + 기록”으로 연결
3. “Preview/Compare/Conflict”는 Plan 탭에 격리(인지부하 분리)

---

### Component Specs (요약 스펙)

#### (A) Global Control Bar (상단 고정)

* Trip/TR/Date(UTC) + Local 표시 + View(섹션 점프)
* 상태 칩: Live/History/Approved/Compare (작업 모드) ([trdash.vercel.app][1])
* 리스크 필터(없음/전체/기상/자원/허가 …)는 **“적용 중 필터 칩”**으로 2차 노출 ([trdash.vercel.app][1])

#### (B) Shift Brief Card

* KPI 3개만(지연/48h/다음 결정) + 즉시 조치 Top3
* “다음 행동(CTA)” 버튼: “결정 확정”, “작업 배정”, “인수인계 기록”

#### (C) Alerts Triage (운영용)

* Severity(색 외에도 아이콘/텍스트), Owner, Due, Last updated, Evidence link
* “GO/NO-GO”는 alert가 아니라 **Decision 객체**로 승격

#### (D) Schedule Table (Plan용)

* 테이블은 “스캔/비교” 중심으로: 정렬/고정 헤더/행 강조(선택일)
* “선택일이 모든 항차 구간 밖” 같은 빈 상태 메시지는 **해결 버튼**을 포함(예: “활성 항차로 자동 이동”) ([trdash.vercel.app][1])

#### (E) Gantt

* 로딩 실패/지연 시 “간이 모드(텍스트 타임라인)” 폴백 제공(운영 화면에서 빈 Gantt는 리스크) ([trdash.vercel.app][1])
* 드래그 이동은 유지하되, “날짜 이동”을 키보드/버튼으로도 제공(WCAG 2.2 Dragging Movements 대응)

### Accessibility Notes (WCAG 2.2 AA 중심)

WCAG 2.2는 **Focus Not Obscured(2.4.11, AA), Dragging Movements(2.5.7, AA), Target Size(2.5.8, AA), Consistent Help(3.2.6), Redundant Entry(3.3.7), Accessible Authentication(3.3.8, AA)** 등 운영 대시보드에서 자주 걸리는 SC를 추가했습니다. ([W3C][4])

**우선 점검 포인트**

* **포커스 가림 방지(2.4.11)**: 상단 고정 바/토스트/드로어가 키보드 포커스를 가리지 않도록(특히 Command palette, Drawer) ([W3C][4])
* **드래그 대체(2.5.7)**: Gantt 드래그 이동 외에 “+1일/-1일”, “다음 윈도우로 이동” 같은 단일 포인터 조작 제공 ([W3C][4])
* **타깃 크기(2.5.8)**: 필터/탭/뷰 토글이 촘촘할수록 위험(특히 dense UI에서) ([W3C][4])
* **일관된 도움말(3.2.6)**: 도움말 위치/표현을 모든 모드에서 동일하게(현재 “도움말” 존재) ([W3C][4])
* **중복 입력(3.3.7)**: Trip/TR/Date 변경 시 동일 정보 재입력 최소화(상단 Control bar 상태 보존) ([W3C][4])

## 7) Prototype Pipeline + Simulation Report

### Prototype Pipeline (권장)

* **Stage A (정보 구조)**: Monitor/Plan/Verify/Export 4탭 IA + Decision Center 상단 고정 와이어
* **Stage B (상호작용)**: Schedule 변경(단건/벌크/드래그) → Preview/Conflict → 승인 → 반영
* **Stage C (접근성)**: 키보드-only로 동일 태스크 수행 가능 확인

### Simulation log (재현 시나리오)

**Scenario 1 — “오늘의 결정 확정”**

* Steps

  1. 상단에서 Selected Date 확인(UTC) ([trdash.vercel.app][1])
  2. Alerts triage에서 “해상 이송 Go/No-Go / 결정: GO / 태그 / 갱신” 확인 ([trdash.vercel.app][1])
  3. Notes & Decisions에 근거 기록 확인(Weather window 등) ([trdash.vercel.app][1])
* Output(측정)

  * Time-to-Decision, 근거 링크 접근 단계 수
* Verdict

  * **Revise**: 결정 확정/승인/기록이 하나의 흐름으로 연결되지 않아 “정보 확인”에서 멈추기 쉬움

**Scenario 2 — “선택일이 항차 구간 밖일 때 복구”**

* Steps

  1. Schedule 영역 진입 → 경고 문구 확인(“선택일이 모든 항차 구간 밖…”) ([trdash.vercel.app][1])
  2. “활성 항차로 이동/오늘” 버튼(문구)로 복구 시도 ([trdash.vercel.app][1])
* Output

  * 사용자가 혼란 없이 “활성 항차”로 점프하는지
* Verdict

  * **Revise**: 빈 상태는 좋지만, “왜 이런 상태인지 + 자동 복구”의 강도가 더 필요

**Scenario 3 — “Gantt 로딩 지연”**

* Steps

  1. Gantt 섹션 진입 → “Loading Gantt…” 확인 ([trdash.vercel.app][1])
* Output

  * p95 로딩 시간, 폴백 존재 여부
* Verdict

  * **Revise**: 운영 화면이라면 폴백/스켈레톤/간이 모드 필요

### Feasibility review

* **Pass(부분) / Revise(핵심)**

  * 기능 구성은 이미 충분히 강함(Preview/Compare/Export/Readiness가 설계에 포함). ([GitHub][5])
  * 다만 “결정→승인→기록”의 엔드투엔드 UX가 1st viewport에서 명확하지 않아 운영 효율 최적화가 필요.

## 8) Validation Plan + Decision Gates

### Validation Plan (권장 테스트)

1. **5-user Task Test**(Shift Lead 3, Planner 2)

   * 과제: “오늘 결정 확정”, “벌크 변경+충돌 해결”, “인수인계 export”
2. **A11y Smoke**

   * Keyboard-only, Focus not obscured, Drag 대체, Target size 점검 (WCAG 2.2 AA) ([W3C][4])
3. **Perf 측정/회귀**

   * Vercel Speed Insights로 Core Web Vitals 기반 실측 데이터를 확보(운영 대시보드는 체감 성능이 곧 신뢰) ([Vercel][7])

### Decision Gates (Go / Revise / Stop)

* **Go**: KPI 측정 이벤트가 박혀 있고(Time-to-Decision 등), A11y blockers 0, 핵심 Flow 회귀 0
* **Revise**: 결정 흐름이 정보 확인에서 끊기거나, Gantt 지연이 잦거나, time model(UTC/Local) 혼재로 오류 가능
* **Stop**: 접근성 차단(키보드로 결정/변경 불가), 승인 없는 파괴적 변경(운영데이터), 데이터 최신성 판단 불가

## 9) Differentiation + Risks + Mitigation

### Differentiation (이 제품이 강해질 수 있는 포인트)

* **“Decision-grade dashboard”**: KPI/일정/리스크를 “결정 객체(승인+근거+기록)”로 묶는 순간 경쟁력이 생김(운영/감사 대응).
* **Scenario/Compare 기반 운영 시뮬레이션**: baseline vs compare를 단순 시각화가 아니라 “승인 가능한 변경 세트”로.

### Risks + Mitigation

* **권한/승인 리스크** → 변경 적용 전 승인(사람-in-the-loop) UX를 기본값으로(특히 bulk/patch/full). Vercel AI SDK 6의 tool execution approval 개념은 이 요구와 정합적입니다. ([Vercel][6])
* **데이터 최신성 리스크** → 모든 카드/섹션에 “last updated / update cadence / stale flag” 표준화(현재는 일부만 노출) ([trdash.vercel.app][1])
* **성능 리스크** → Speed Insights 기반 측정-개선 루프(최소한 p95 상호작용) ([Vercel][7])
