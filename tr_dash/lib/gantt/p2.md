## 3) Deep Insights: 개선 포인트를 “의사결정 루프” 기준으로 재구성

아래는 단순 UI 폴리시가 아니라, **현장 운영(Shift)에서 실제로 시간을 먹는 지점**을 기준으로 한 개선 제안입니다.

### Insight A) Shift Brief를 “요약 텍스트”에서 “결정 카드(Decision Card)”로 격상

**현상**

* Shift Brief에 “48시간 내 중요 작업 8건”, “다음 결정 창 18:00 현지” 등 핵심 정보가 있지만, 그 아래 섹션들과 연결이 약합니다. ([TR Transport Dashboard][1])

**개선 제안**

* Shift Brief를 아래 4요소로 고정(상단 고정/Sticky 가능):

  1. **오늘의 1~3개 결정을 카드화** (예: “Go/No-Go”, “MZP Slot Confirm”, “SPMT shift change”)
  2. 각 카드에 **근거 신호(Weather/Permit/Resource) + 임계치/조건 + 신뢰도(Confidence)**
  3. **Owner + Due time + Escalation path**
  4. “결정하기” 버튼 → Evidence/History 탭에서 자동으로 해당 결정 템플릿 열기

**효과**

* 사용자는 아래로 스크롤하며 퍼즐 조립을 하지 않고, **카드에서 바로 행동**합니다(대시보드 정의에 부합). ([Nielsen Norman Group][2])

---

### Insight B) Go/No-Go는 “상태(결정: GO)”가 아니라 “근거/조건/만료”가 핵심

**현상**

* Alerts Triage에 “해상 이송 Go/No-Go 결정: GO”, 트리거(WX_WAVE, WX_WIND), 갱신일(2026-02-01)이 보이지만, 운영자가 궁금해하는 “그래서 기준이 뭐고, 언제까지 유효한가?”가 빠져 있습니다. ([TR Transport Dashboard][1])

**개선 제안**

* Go/No-Go를 아래 스키마로 표준화(각각 UI에 노출):

  * **Decision status**: GO / HOLD / NO-GO
  * **Validity window**: `valid_from ~ valid_until`
  * **Thresholds**: (예) wind ≤ X kt, wave ≤ Y m, visibility ≥ Z …
  * **Latest forecast snapshot timestamp** (데이터 최신성)
  * **Required evidence** (PTW, RA, checklist) 충족 여부 → Evidence Gate와 연결

Evidence Gate는 이미 설계/테스트가 존재하므로, “Go/No-Go 카드”가 **state-machine/evidence-gate 결과를 요약해 보여주도록** 붙이는 것이 자연스럽습니다. 【238:14†SYSTEM_ARCHITECTURE.md†L1-L15】

### Insight C) Alerts Triage는 “분류 탭”보다 “우선순위·라우팅·에스컬레이션”이 성패

**현상**

* “알림 분류(Alerts Triage)”가 존재하지만, 현재 노출 정보는 카테고리(기상/해상/SPMT/안전/항만) 중심입니다. ([TR Transport Dashboard][1])

**개선 제안 (운영 관점)**

* 운영 알람은 SRE 관점에서도 “Severity(심각도), Routing(담당), Escalation(미확인 시 승격)”이 핵심입니다. ([pragmaticsre.com][3])verity 레벨(예: Critical / High / Medium / Info)**

  * **Ack(확인) + Snooze(유예) + Escalate(승격)** 액션
  * “이 알람이 영향을 주는 Trip/Activity/결정” 자동 링크
  * 알람 폭주 방지를 위한 **그룹핑(동일 원인/근접 시간)**

**기술적 연결**

* WhyPanel의 “root cause + suggested action”이 이미 구조적으로 존재하므로, Alerts에서 곧장 WhyPanel로 연결되게 하면 **Alert → Root cause → Suggested action → Preview → Apply**가 하나의 루프로 닫힙니다. 【242:6†LAYOUT.md†L1-L22】

### Insight D) “Selected Date”는 UX 상 ‘정렬축’이므로, 초기값/타임존 정책이 제품 신뢰를 좌우

**현상**

* Selected Date가 `Jan 26, 2026`이고, Schedule 표에서 “선택일이 모든 항차 구간 밖입니다” 메시지가 뜹니다. ([TR Transport Dashboard][1])
* 또한 Shift Brief는 “18:00 현지”처럼 로컬을 쓰는데, 필터는 “날짜(UTC)”로 표기됩니다. ([TR Transport Dashboard][1])

**개선 제안**

* **타임존 정책을 제품 레벨에서 명문화**

  * “데이터 계산/저장 = UTC”, “표시 = 현지(토글 가능)” 같은 단순한 규칙을 UI에서 명시
* 초기 Selected Date는 이미 `getSmartInitialDate()`로 “면 가장 가까운 voyage start” 전략을 갖고 있습니다. 【238:8†WORK_LOG_20260202.md†L37-L41】

  * 그런데 UI에서 “선택일이 항차 밖”이 뜨는 순간 사용자는 불신합니다.
  * **초기 선택일의 결과를 상단에 설명**: “오늘(현지) 기준으로 가장 가까운 항차 시작일로 이동했습니다” 같은 한 줄이 큰 효과.

### Insight E) History/Evidence/Compare는 ‘기능’이 아니라 ‘감사(Audit) 체계’ — localStorage는 한계가 명확

**현상**

* UI 하단에 “History Evidence Compare Diff Trip Closeout”가 보이지만, “Load SSOT to display history” 같은 문구로 **사용자에게 ‘비어 있음’을 노출**하고 있습니다. ([TR Transport Dashboard][1])
* 문서상 History/Evidence는 localStorage 기반(append-only)로 구현되어 있습니다. 【242:11†README서 History/Evidence는 ‘나중에 보는 기록’이 아니라 ‘결정의 일부’**입니다.
* 우선순위:

  1. “Load SSOT…” 같은 빈 상태 문구를 없애고, **초기 로딩 시 기본 SSOT를 즉시 결합**(또는 샘플/데모 데이터를 명시적으로 표시)
  2. localStorage는 “개인 메모”까지만, **결정/증빙/승인은 서버 영속(append-only log)** 권장
  3. Compare Diff는 이미 Baseline/Compare as-of 표기까지 개선되어 있으므로【238:13†README.md†L22-L24】, “Diff 요약(Δ완료일/ΔCritical path/ΔGo-NoGo 영향)”을 KPI에 끌어올리면 가치가 급상승합니다.
